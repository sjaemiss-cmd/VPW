# 딸깍연구소 브랜드 아이덴티티 가이드라인

**Brand Identity Guidelines v1.0**

> 이 문서는 딸깍연구소(Click Lab)의 브랜드 전략, 비주얼 아이덴티티, 커뮤니케이션 가이드라인을 정의합니다.
> 로고 제작, 채널 에셋 디자인, 콘텐츠 제작 시 이 문서를 기준으로 일관된 브랜드 경험을 구현합니다.

---

## 1. 브랜드 에센스 (Brand Essence)

### 1.1 핵심 컨셉

**"기술의 민주화를 유머로 증명한다."**

딸깍연구소는 AI 바이브코딩이라는 새로운 기술 패러다임을 '진지한 교육'이 아닌 '엉뚱한 실험'으로 풀어내는 채널이다. 코딩을 모르는 사람도 AI에게 자연어로 지시하면 앱을 만들 수 있다는 사실을, 세상에서 가장 쓸데없는 앱을 만들어봄으로써 증명한다. 핵심은 **아이러니** --- 쓸모없는 것을 진지하게 만드는 과정 자체가 기술 민주화의 가장 강력한 증거가 된다.

### 1.2 브랜드 미션 (Mission)

> "누구나 딸깍 한 번이면 아이디어를 현실로 만들 수 있다는 것을, 가장 쓸데없는 앱으로 증명합니다."

- **누구나**: 코딩 지식 불필요, 진입 장벽 제거
- **딸깍 한 번**: AI 바이브코딩의 간편함, 채널명과의 직결
- **가장 쓸데없는 앱**: 예능적 재미, 실험 정신, 반전의 매력

### 1.3 브랜드 비전 (Vision)

> "바이브코딩 예능의 대명사. '하찮은 문제'에서 시작해 누구나 만드는 사람(Creator)이 되는 세상."

장기적으로 딸깍연구소는 단순한 유튜브 채널을 넘어, AI 시대에 '만드는 즐거움'을 대중에게 전파하는 브랜드로 성장한다.

### 1.4 브랜드 가치 (Values)

| 가치 | 정의 | 행동으로의 발현 |
|------|------|-----------------|
| **엉뚱함 (Absurdity)** | 일상의 하찮은 문제를 진지하게 다루는 유쾌한 반전 | 매 에피소드 "이걸 앱으로 만들면 되는 거 아닌가?"로 시작 |
| **투명함 (Transparency)** | 시행착오, 에러, 실패를 숨기지 않는 정직한 과정 공개 | 바이브코딩 과정을 편집 없이 보여주고, 에러도 콘텐츠로 활용 |
| **접근성 (Accessibility)** | 기술을 모르는 사람도 이해하고 즐길 수 있는 쉬운 언어 | 전문 용어 대신 일상 언어 사용, 시청자 누구나 따라할 수 있는 구조 |
| **실행력 (Execution)** | 말로만 끝나지 않고 매 에피소드 실제 작동하는 결과물 도출 | 완성된 앱의 QR코드 공유, 실사용 시연 |

### 1.5 브랜드 퍼스낼리티 (Brand Personality)

딸깍연구소의 문화원형(Cultural Archetype)은 **Jester(광대) x Creator(창조자)**의 교차점이다.

| 성격 특성 | 설명 | 표현 방식 |
|-----------|------|-----------|
| **무심한 듯 진지한** | 쓸데없는 것을 대단한 프로젝트처럼 진지하게 접근 | "이건 인류에게 꼭 필요한 앱입니다" (진지한 얼굴로) |
| **호기심 가득한** | "이거 되나?" 하는 순수한 실험 정신 | 매번 새로운 하찮은 문제에 진심으로 몰입 |
| **솔직하고 허술한** | 완벽하지 않은 과정을 그대로 보여주는 인간적 매력 | 에러 앞에서 당황하는 리액션, 꾸미지 않은 반응 |
| **은근히 똑똑한** | 유머 속에 실제로 작동하는 기술과 인사이트가 숨어 있음 | 웃기지만 결국 앱은 진짜 완성됨 |

**한 문장 퍼스낼리티 정의:**
> "친구 중에 제일 엉뚱한데, 막상 시키면 진짜 해오는 그 친구."

### 1.6 브랜드 약속 (Brand Promise)

> "딸깍연구소를 보면, 웃기면서도 '나도 만들어볼까?'라는 생각이 들게 합니다."

- 시청자에게는: 매 에피소드 웃음 + "나도 할 수 있겠다"는 영감
- 크리에이터로서는: 매 에피소드 실제로 작동하는 앱 결과물 보장

## 2. 브랜드 보이스 & 톤 (Brand Voice & Tone)

### 2.1 보이스 특성 (Voice Characteristics)

딸깍연구소의 보이스는 **세 가지 축**으로 정의된다.

#### 축 1: 캐주얼하되 무례하지 않은 (Casual, Not Rude)
- 친구에게 말하듯 편한 반말 톤
- 그러나 시청자를 무시하거나 비하하는 표현은 절대 사용하지 않음
- 기본 호칭: "여러분" (격식 없이 자연스럽게)

#### 축 2: 엉뚱하되 무지하지 않은 (Absurd, Not Ignorant)
- 하찮은 소재를 다루지만 기술적 내용은 정확하게 전달
- "몰라서 웃기는 것"이 아니라 "알면서도 일부러 엉뚱하게 하는 것"
- AI, 코딩 관련 정보는 틀리지 않도록 팩트체크 필수

#### 축 3: 자조적이되 비관적이지 않은 (Self-deprecating, Not Pessimistic)
- "코딩 1도 모르는 제가" 같은 자조적 유머 사용
- 그러나 결론은 항상 긍정적: "근데 이거 진짜 됐잖아요?"
- 실패도 즐기는 태도, 과정 자체를 콘텐츠로 전환

### 2.2 톤 변주 (Tone Variations)

하나의 보이스 아래, 상황에 따라 톤을 조절한다.

| 상황 | 톤 | 예시 |
|------|-----|------|
| **에피소드 인트로** | 진지한 척 장난스러운 | "오늘의 하찮은 문제... 이건 진짜 심각합니다." |
| **바이브코딩 과정** | 호기심 + 약간 흥분 | "어 이거 되나? 잠깐 이거 진짜 되는 거야?" |
| **에러/시행착오** | 자조적 + 코믹 | "아 Claude 왜 이래... 나한테 왜 이러는 거야" |
| **결과물 완성** | 진심 감탄 + 허탈 | "된다! 야 이거 진짜--- 근데 이걸 누가 쓰지?" |
| **커뮤니티 소통** | 친근한 + 참여 유도 | "다음에 만들 하찮은 앱, 여러분이 정해주세요" |
| **채널 공식 소개** | 위트 있는 자기소개 | "코딩 모르는 대학생이 AI로 쓸데없는 앱 만드는 채널" |

### 2.3 시그니처 언어 (Signature Language)

반복 사용하여 브랜드 인지도를 높이는 핵심 표현들:

| 용도 | 시그니처 표현 |
|------|---------------|
| **에피소드 트리거** | "이걸 앱으로 만들면 되는 거 아닌가?" |
| **바이브코딩 시작** | "대딸깍의 시대, 제가 해결해보겠습니다" |
| **에피소드 클로징** | "다음 하찮은 문제에서 만나요" |
| **슬로건** | "딸깍 한 번이면 앱이 뚝딱, 코딩은 AI가 합니다" |
| **쓸모 판정** | "쓸모 판정: X점 / 5점" |
| **인트로 터미널** | `$ 오늘의 하찮은 문제:` |

### 2.4 DO & DON'T

#### DO (이렇게 합니다)

- 일상 언어로 기술을 설명한다
  - O: "AI한테 '이거 만들어줘' 하면 코드가 쫙 나와요"
- 자조적 유머를 사용한다
  - O: "저 코딩 1도 모릅니다. 진짜로."
- 시행착오를 솔직하게 보여준다
  - O: "에러 5번 나서 멘탈 나갈 뻔했는데요..."
- 하찮은 것을 과대포장한다 (의도적 아이러니)
  - O: "인류 최대의 과제를 해결하겠습니다: 점심 메뉴 고르기"
- 시청자를 참여시킨다
  - O: "여러분의 하찮은 문제를 제보해주세요"

#### DON'T (이렇게 하지 않습니다)

- 전문 용어를 설명 없이 사용하지 않는다
  - X: "REST API 엔드포인트를 설정하고 OAuth 인증을..."
- 기술을 과시하거나 시청자를 깔보지 않는다
  - X: "이것도 모르세요? 기본인데..."
- 다른 채널이나 크리에이터를 비하하지 않는다
  - X: "XX 채널은 너무 재미없죠"
- 특정 성별, 연령, 직업군을 비하하는 유머를 사용하지 않는다
  - X: (성별 고정관념에 기반한 소재 사용)
- 사이비 과학이나 검증되지 않은 기술 정보를 유포하지 않는다
  - X: "AI가 곧 인류를 지배할 겁니다" (공포 마케팅)
- 결과물의 완성도를 과장하지 않는다
  - X: "전문 개발자 수준의 앱이 나왔습니다" (실제로 하찮은 앱인데)

### 2.5 플랫폼별 톤 가이드

| 플랫폼 | 톤 조절 | 예시 |
|--------|---------|------|
| **유튜브 롱폼** | 서사적, 기승전결 구조, 풍부한 리액션 | 5단계 포맷 구조에 따른 톤 변주 활용 |
| **유튜브 쇼츠** | 임팩트 우선, 3초 이내 훅, 빠른 전환 | "이 앱 실화임?" + 결과물 바로 보여주기 |
| **커뮤니티 탭** | 참여 유도, 투표/제보, 친근함 극대화 | "다음 에피소드 소재 골라주세요" |
| **영상 제목** | 클릭 유도 + 채널 톤, 과장법 활용 | "헤어진 여자친구 절대 안 마주치게 해주는 앱 만들어봤습니다" |
| **영상 설명란** | 정보 정리 + 캐주얼 어투, 링크 제공 | 앱 QR, 사용 도구, 타임라인 등 |

## 3. 비주얼 아이덴티티 (Visual Identity)

### 3.1 디자인 철학

딸깍연구소의 비주얼은 **"에디토리얼 터미널"** 이라는 하나의 키워드로 수렴한다.

- **에디토리얼(Editorial)**: 매거진처럼 정돈된 레이아웃, 깔끔한 타이포그래피, 세련된 여백
- **터미널(Terminal)**: 개발자 코드 에디터/터미널의 다크 테마, 모노스페이스 폰트, 네온 컬러

이 두 요소의 결합이 딸깍연구소만의 비주얼 정체성을 만든다. 채널이 다루는 '하찮은 앱'과 비주얼의 '세련됨' 사이의 갭(Gap)이 의도적 아이러니를 형성하며, 이것이 Jester x Creator 아키타입의 시각적 표현이다.

**비주얼 원칙:**
1. **다크 퍼스트(Dark First)**: 모든 에셋은 다크 배경을 기본으로 설계한다
2. **코드 모티프(Code Motif)**: 터미널, 코드 에디터의 UI 요소를 그래픽 모티프로 활용한다
3. **네온 액센트(Neon Accent)**: 어두운 배경 위에 네온 그린/옐로우가 에너지와 생동감을 부여한다
4. **정보 위계(Information Hierarchy)**: 복잡한 정보도 명확한 위계를 통해 직관적으로 전달한다

#### 3.2.2 글로우 효과 팔레트 (Glow Effects)

네온 느낌을 극대화하는 반투명 글로우 컬러. 배경 하이라이트, 박스 섀도우, 호버 효과에 사용한다.

```
--glow-green     rgba(57, 211, 83, 0.15)     그린 글로우 (호버, 강조)
--glow-green-s   rgba(57, 211, 83, 0.25)     그린 글로우 강한 버전
--glow-yellow    rgba(240, 192, 0, 0.10)     옐로우 글로우
--glow-pink      rgba(247, 120, 186, 0.10)   핑크 글로우
--glow-blue      rgba(88, 166, 255, 0.10)    블루 글로우
```

#### 3.2.3 컬러 사용 규칙

**비율 원칙 (70-20-10 Rule):**
- **70%** --- 다크 배경 (#0d1117 ~ #1c2128): 콘텐츠의 무대
- **20%** --- 텍스트 + 보더 (#e6edf3, #8b949e, #30363d): 정보 전달
- **10%** --- 액센트 컬러 (#39d353, #f0c000, #f778ba): 시선 집중

**액센트 컬러 우선순위:**
1. **네온 그린 (#39d353)**: 가장 많이 사용. 채널의 시그니처 컬러. CTA, 강조, 로고에 사용
2. **옐로우 (#f0c000)**: 두 번째 빈도. 아이디어, 경고, 보조 강조에 사용
3. **핑크 (#f778ba)**: 포인트로만 사용. 코드 키워드, 반전 순간, 에러 강조에 사용
4. **블루 (#58a6ff)**: 최소 사용. 링크, 정보성 요소에만 사용

**접근성 (Accessibility):**
- 다크 배경(#0d1117) 위 텍스트(#e6edf3): 대비율 약 13.5:1 --- WCAG AAA 충족
- 다크 배경(#0d1117) 위 네온 그린(#39d353): 대비율 약 8.2:1 --- WCAG AAA 충족
- 다크 배경(#0d1117) 위 보조 텍스트(#8b949e): 대비율 약 4.6:1 --- WCAG AA 충족
- 모든 액센트 컬러는 다크 배경 위에서 단독으로도 가독성 유지

**금지 조합:**
- 액센트 컬러끼리 직접 겹치지 않는다 (그린 텍스트 위에 핑크 등)
- 밝은 배경(#e6edf3) 위에 액센트 컬러를 텍스트로 사용하지 않는다 (대비 부족)
- 네온 그린을 넓은 면적의 배경색으로 사용하지 않는다 (눈의 피로)

### 3.3 타이포그래피 시스템 (Typography System)

#### 3.3.1 서체 선정

```
한글 본문/제목
━━━━━━━━━━━━━━━━━━━━━━
Noto Sans KR (Google Fonts)
- 사용 웨이트: 400(Regular), 500(Medium), 700(Bold), 900(Black)
- 역할: 제목, 본문, UI 텍스트 전반
- 선정 이유: 넓은 웨이트 레인지, 우수한 가독성, 한글/영문 혼용 안정성
- 대체 서체: Pretendard (로컬 환경에서 사용 가능할 때)

영문/코드 전용
━━━━━━━━━━━━━━━━━━━━━━
JetBrains Mono (Google Fonts)
- 사용 웨이트: 400(Regular), 500(Medium), 700(Bold)
- 역할: 코드 표현, 터미널 UI 모티프, 영문 라벨, 숫자
- 선정 이유: 코드 에디터 네이티브 느낌, 리가처(ligature) 지원,
             개발자 커뮤니티에서의 높은 인지도
- 대체 서체: Fira Code
```

#### 3.3.2 타이포그래피 위계 (Type Hierarchy)

영상 내부, 썸네일, 웹 에셋 등 모든 터치포인트에 적용하는 텍스트 위계:

```
Level 1 — 대제목 (Display)
서체: Noto Sans KR Black (900)
크기: 40~64px (매체에 따라 조절)
행간: 1.2
용도: 썸네일 메인 카피, 에피소드 타이틀 카드
예시: "점심 메뉴 골라주는 앱"

Level 2 — 섹션 제목 (Heading)
서체: Noto Sans KR Bold (700)
크기: 28~36px
행간: 1.3
용도: 영상 내 섹션 구분, 채널 소개 제목
예시: "오늘의 하찮은 문제"

Level 3 — 소제목 (Subheading)
서체: Noto Sans KR Medium (500)
크기: 18~24px
행간: 1.4
용도: 카드 제목, 기능 설명 제목
예시: "바이브코딩 시작"

Level 4 — 본문 (Body)
서체: Noto Sans KR Regular (400)
크기: 14~16px
행간: 1.8
용도: 설명 텍스트, 자막, 영상 설명란
예시: 일반 나레이션 자막

Level M — 모노스페이스 (Monospace)
서체: JetBrains Mono Regular/Bold
크기: 매체에 따라 가변 (본문 대비 90% 크기 권장)
행간: 1.6
용도: 코드 표현, 터미널 프롬프트, 기술 라벨, 섹션 태그
예시: "$ 오늘의 하찮은 문제:", "BUILD SUCCESS"
```

#### 3.3.3 타이포그래피 사용 규칙

- **한글 제목에는 Noto Sans KR 700 또는 900만 사용한다.** Regular(400)로 제목을 쓰지 않는다.
- **코드/터미널 관련 텍스트는 반드시 JetBrains Mono를 사용한다.** 한글 폰트로 코드를 표현하지 않는다.
- **한 화면에 3가지 이상의 서체 크기를 사용하지 않는다.** 위계가 혼란스러워진다.
- **자막은 Noto Sans KR 700 + 배경 하이라이트(그린 또는 반투명 블랙)** 조합을 기본으로 한다.
- **숫자는 JetBrains Mono를 사용한다.** 타임코드, 점수, 통계 등 숫자가 중심인 정보에 적용.

### 3.4 그래픽 모티프 (Graphic Motifs)

딸깍연구소의 비주얼을 구성하는 반복 요소들:

#### 3.4.1 터미널 프롬프트 `$_`

채널의 핵심 그래픽 심볼. 모든 에셋에 반복 등장하여 브랜드 인지도를 높인다.

```
활용 방식:
- 인트로 타이핑 애니메이션의 시작점
- 섹션 구분자로 활용: $ section_name
- 로고 심볼의 핵심 요소
- 썸네일 코너 장식
- 커뮤니티 탭 게시물 시작 부분
```

#### 3.4.2 코드 에디터 UI

맥OS 스타일의 코드 에디터 창 모티프. 상단에 빨강/노랑/초록 세 개의 원형 버튼(Traffic Lights)과 파일명 탭이 있는 형태.

```
구성 요소:
- 트래픽 라이트: ● ● ● (빨강 #ff5f57, 노랑 #febc2e, 초록 #28c840)
- 파일명 탭: JetBrains Mono로 파일명 표기
- 라인 넘버: 좌측 여백에 줄번호
- 신택스 하이라이팅: 핑크(키워드), 그린(문자열), 옐로우(함수), 블루(변수)
```

활용처: 영상 내 정보 카드, 웹 에셋의 콘텐츠 블록, 썸네일 내 텍스트 박스

#### 3.4.3 글리치/스캔라인 효과 (Glitch/Scanline)

디지털/코딩 느낌을 강조하는 보조 효과. 과도하게 사용하면 가독성을 해치므로 트랜지션이나 강조 순간에만 제한적으로 사용한다.

```
사용 규칙:
- 타이틀 시퀀스 전환 시: 0.3~0.5초 이내의 글리치 트랜지션
- 에러 발생 순간: 화면 전체에 잠깐 글리치 이펙트
- 정적 에셋(배너 등)에서는 미세한 스캔라인 텍스처만 적용
- 글리치를 3초 이상 지속하지 않는다
```

#### 3.4.4 그리드/도트 패턴

배경 텍스처로 사용하는 미세한 도트 그리드 또는 가느다란 격자선. 기술적 정밀함의 느낌을 부여한다.

```
사용 규칙:
- 불투명도 5~10% 이하로 은은하게 적용
- 배경 전체 또는 카드 내부에 텍스처로 사용
- 도트 간격: 20~40px
- 컬러: --border (#30363d) 또는 그보다 어두운 톤
```

#### 3.4.5 네온 글로우 라인

구분선, 카드 상단 보더, 텍스트 하이라이트 등에 사용하는 네온 발광 효과.

```
사용 규칙:
- 그린 글로우를 기본으로 사용
- 그라디언트 형태: transparent → accent-green → transparent
- 두께: 1~2px (얇게)
- 카드 상단 호버 시 나타나는 등 인터랙션 요소로 활용
- 면적이 넓은 글로우는 불투명도를 낮춰 은은하게 처리
```

## 4. 로고 디자인 방향 (Logo Design Direction)

### 4.1 로고에 담아야 할 의미

딸깍연구소 로고는 다음 세 가지 의미를 시각적으로 전달해야 한다:

1. **"딸깍" --- 클릭/실행의 순간**: 버튼을 누르는 행위, AI에게 명령을 내리는 순간, 마우스 클릭의 촉각적 느낌
2. **"연구소" --- 실험과 탐구**: 진지한 척하는 유머, 체계적인 엉뚱함, 실험실의 격식
3. **코드/터미널 --- 기술적 기반**: 바이브코딩의 세계, 개발자 문화의 시각적 언어

### 4.2 로고 형태 방향 (Logo Architecture)

3가지 버전의 로고 시스템을 구축한다:

#### A. 콤비네이션 마크 (Combination Mark) --- 메인 로고

심볼 + 워드마크의 결합. 채널의 공식 대표 로고로 가장 많이 사용한다.

```
구조 (가로형):
[심볼]  딸깍연구소
        Click Lab

구조 (세로형):
    [심볼]
  딸깍연구소
   Click Lab
```

- **심볼**: 터미널 프롬프트 `$_`를 모티프로 한 아이콘 (상세는 4.3 참조)
- **워드마크 "딸깍연구소"**: Noto Sans KR Black(900), 자간 약간 조임(-2% 내외)
- **서브텍스트 "Click Lab"**: JetBrains Mono Regular(400), 작은 크기, --text-secondary 컬러

#### B. 심볼 마크 (Symbol Mark) --- 프로필/파비콘용

심볼만 단독으로 사용하는 버전. 프로필 이미지, 워터마크, 앱 아이콘 등 작은 공간에 사용한다.

```
형태: 정사각형 또는 원형에 담기는 심볼
최소 크기: 32x32px (디지털), 10mm (인쇄)
```

#### C. 워드마크 (Wordmark) --- 텍스트 전용

심볼 없이 채널명 텍스트만으로 구성된 버전. 자막, 엔드카드, 설명란 등 텍스트 중심 환경에서 사용한다.

```
형태: 딸깍연구소 (한 줄)
서체: Noto Sans KR Black(900)
특징: "딸" 또는 "$" 부분에 네온 그린 컬러 적용으로 심볼 없이도 브랜드 인지
```

### 4.3 터미널 프롬프트 `$_` 모티프 활용 방안

로고 심볼의 핵심은 터미널 프롬프트 기호 `$_`의 창의적 재해석이다.

#### 방향 1: `$` 기호의 변형

- `$` 기호를 기하학적으로 단순화하여 심볼로 사용
- `$` 안에 클릭/실행을 의미하는 시각 요소(전원 버튼, 커서 등)를 결합
- 네온 그린 컬러로 터미널 커서가 깜빡이는 느낌 부여

#### 방향 2: `$_` 프롬프트 라인 자체를 로고화

- `$_` 또는 `$ █` (블록 커서 포함)를 워드마크 앞에 배치하여 터미널 입력 라인처럼 보이게 구성
- 예시: `$ 딸깍연구소_` --- 타이핑 중인 느낌, 진행 중인 실험의 뉘앙스
- 애니메이션 버전에서는 커서 깜빡임 효과 적용 가능

#### 방향 3: 코드 에디터 프레임 + 텍스트

- 트래픽 라이트(● ● ●)가 있는 미니 코드 에디터 창 안에 채널명을 넣는 형태
- 파일 탭에 "click_lab.exe" 또는 "딸깍.app" 같은 표기
- 가장 장식적인 버전이므로 특정 에셋(배너, 인트로)에만 사용

#### 권장 방향

**방향 2를 메인으로 채택하되, 방향 1의 `$` 단독 심볼을 보조로 활용하는 것을 권장한다.** 이유:

- 방향 2는 가장 직관적으로 "터미널/코딩" 느낌을 전달하면서도, 브랜드 네임과 자연스럽게 결합됨
- 방향 1의 `$` 심볼은 프로필 이미지, 파비콘 등 소형 에셋에 독립적으로 사용 가능
- 방향 3은 복잡도가 높아 축소 시 가독성이 떨어지므로 메인 로고보다는 배너/인트로용으로 한정

### 4.4 로고 컬러 버전

```
1. 기본 버전 (Primary)
   - 다크 배경(#0d1117) 위에 화이트(#e6edf3) 텍스트 + 네온 그린(#39d353) 액센트
   - 가장 빈번하게 사용하는 공식 버전

2. 모노크롬 화이트 (White Mono)
   - 다크 배경 위에 전체 화이트(#e6edf3)
   - 영상 내 워터마크, 액센트가 방해되는 상황에서 사용

3. 모노크롬 다크 (Dark Mono)
   - 밝은 배경 위에 전체 다크(#0d1117)
   - 인쇄물, 공식 문서, 밝은 배경이 불가피한 환경에서 사용

4. 네온 버전 (Neon)
   - 다크 배경 위에 전체 네온 그린(#39d353) + 글로우 효과
   - 특별 이벤트, 강조가 필요한 순간에만 제한적 사용
```

### 4.5 로고 클리어 스페이스 (Clear Space)

로고 주변에 다른 요소가 침범하지 않아야 하는 최소 여백.

```
측정 기준: 심볼 마크 높이의 50% (= "X" 단위)

    ┌─── X ───┐
    │         │
X   [$] 딸깍연구소   X
    │         │
    └─── X ───┘

- 로고 상하좌우 모두 X 이상의 여백 확보
- 배경이 복잡한 이미지 위에 올릴 때는 반투명 박스 또는
  충분한 여백으로 가독성 확보
```

### 4.6 로고 최소 크기 (Minimum Size)

```
콤비네이션 마크 (가로형): 최소 너비 120px (디지털) / 30mm (인쇄)
콤비네이션 마크 (세로형): 최소 너비 80px (디지털) / 20mm (인쇄)
심볼 마크 단독:          최소 32x32px (디지털) / 10mm (인쇄)
워드마크 단독:           최소 너비 100px (디지털) / 25mm (인쇄)
```

### 4.7 로고 금지 사항

다음과 같이 로고를 변형하거나 사용하지 않는다:

```
X  비율을 왜곡하여 늘이거나 찌그러뜨리지 않는다
X  지정된 컬러 버전 외의 색상을 임의로 적용하지 않는다
X  로고에 그림자, 3D 효과, 외곽선, 그라디언트를 추가하지 않는다
X  복잡한 패턴이나 사진 위에 직접 올리지 않는다 (배경 박스 필요)
X  로고를 45도 이상 회전하지 않는다
X  클리어 스페이스 안에 다른 텍스트나 그래픽을 넣지 않는다
X  심볼과 워드마크의 상대적 크기 비율을 임의로 변경하지 않는다
X  로고를 장식적 패턴으로 반복 배치하지 않는다

## 5. 채널 에셋 가이드 (Channel Asset Guide)

### 5.1 프로필 이미지 (Profile Image)

유튜브 채널 아이콘으로 사용. 원형으로 크롭되므로 이를 고려하여 디자인한다.

```
사양:
- 크기: 800x800px (유튜브 권장)
- 형태: 정사각형 원본, 원형 크롭 시 핵심 요소 유지
- 포맷: PNG (투명 배경 불가, 유튜브 원형 마스크 적용)

디자인 방향:
- 배경: --bg-primary (#0d1117) 또는 --bg-secondary (#161b22)
- 중앙: '$' 심볼 마크 (네온 그린 #39d353)
- 심볼 주변에 미세한 네온 글로우 효과
- 원형 크롭 시 심볼이 중앙에 오도록 충분한 패딩
- 단순하고 명확하게 — 작은 크기(40x40px)에서도 인식 가능해야 함

금지:
- 프로필에 채널명 전체를 넣지 않는다 (크기가 작아 읽히지 않음)
- 사진이나 실사 이미지를 프로필로 사용하지 않는다 (채널 톤과 불일치)
- 밝은 배경을 사용하지 않는다
```

### 5.2 채널 배너 (Channel Banner)

유튜브 채널 아트. 기기별로 다른 영역이 보이므로 안전 영역을 반드시 준수한다.

```
사양:
- 전체 크기: 2560x1440px (유튜브 권장)
- TV 표시 영역: 2560x1440px (전체)
- 데스크톱 표시 영역: 2560x423px (중앙)
- 모바일 표시 영역: 1546x423px (중앙 핵심)

안전 영역 (모든 기기 공통 표시):
- 1546x423px 영역 안에 핵심 정보 배치

디자인 방향:
- 배경: --bg-primary 베이스, 미세한 도트 그리드 패턴 또는 코드 라인 텍스처
- 중앙 (안전 영역):
  - 콤비네이션 마크 (가로형)
  - 슬로건: "딸깍 한 번이면 앱이 뚝딱, 코딩은 AI가 합니다"
    (JetBrains Mono, --text-secondary, 로고 하단에 배치)
- 좌우 확장 영역 (데스크톱/TV에서만 보임):
  - 터미널 코드 라인 텍스처 (어두운 톤으로 배경처럼)
  - 또는 네온 그린 글로우 라인이 좌우로 뻗어나가는 형태
- 상하 확장 영역 (TV에서만 보임):
  - 다크 배경 유지, 미세한 그리드 패턴

분위기: 
- 코드 에디터를 열었을 때의 첫 화면 느낌
- 터미널에 커서가 깜빡이며 입력을 기다리는 상태
- 절제되고 깔끔하되 "개발자 감성"이 느껴지는 톤

### 6.4 차별화 요약 (Differentiation Summary)

| **1. 바이브코딩 특화** | AI에게 자연어로 시키는 코딩만 다룸. 전통적 코딩 교육 아님 | 코딩애플/노마드코더: 전통 코딩 교육 |
| **2. 예능 포맷** | 5단계 서사 구조 + 리액션 + 시행착오의 예능적 편집 | 개발 채널: 튜토리얼/강의 포맷 |
| **3. 매회 실물 결과물** | 매 에피소드 실제로 작동하는 앱이 완성됨 (QR코드 공유) | 침착맨/피식대학: 토크/상황극 (결과물 없음) |
| **4. 의도적 무용함** | "쓸데없는 앱"이라는 컨셉이 역설적으로 기술 접근성을 증명 | 긱블: 유용하거나 대단한 결과물 지향 |
| **5. 1인 대학생** | 비전문가 시점으로 AI 시대의 기술 민주화를 체현 | 대부분의 개발 채널: 현직 개발자/전문가 |

### 6.5 경쟁 우위 유지 전략

딸깍연구소의 블루오션 포지션을 장기적으로 지키기 위한 브랜드 전략:

1. **"하찮은 앱" 프레임 고수**: 유용한 앱으로 방향 전환하지 않는다. "쓸데없음"이 브랜드의 핵심 자산이다.
2. **비전문가 시점 유지**: 크리에이터가 코딩 실력이 늘어도, "코딩 모르는 사람" 시점을 유지한다. 이것이 시청자 동일시의 핵심이다.
3. **포맷 일관성**: 5단계 구조(문제 발견 - 기획 - 바이브코딩 - 시연 - 판정)를 흔들지 않는다. 포맷의 반복이 브랜드 인지도를 만든다.
4. **시청자 참여 확대**: "하찮은 문제 제보"를 통해 커뮤니티를 브랜드의 일부로 만든다.
5. **비주얼 일관성**: 이 가이드라인에 따른 일관된 비주얼 시스템을 모든 터치포인트에 적용한다.
